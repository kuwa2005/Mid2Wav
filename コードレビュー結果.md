### 不具合・バグの指摘と修正提案

#### 1. CC0（Bank MSB）単独変更でプリセットが再構築されない

`renderToWav` で `hasBankSelectInBlock` を設定するが、`programChange` は PC イベントがある場合のみ呼ばれる。

**修正提案:** `hasBankSelectInBlock` が true のときも `programChange(ch, m_channels[ch].program, useBank)` を呼ぶ。

> **✅ 対応済み** `sf_synth.cpp:929-933` — `hasBankSelectInBlock && !needProgramChange` の場合に `programChange()` を呼ぶ分岐を追加。

#### 2. CC1（Modulation）がレンダリングに反映されない

MIDI パースでは CC1 を記録（`midi_file.cpp` L177）するが、`renderToWav` のブロックループに `expr.modulation` 適用がない。`processVoice` は `m_channels[ch].modulation` を参照するため、ビブラートが常に 0。

**修正提案:** volume/expression と同様に `expr.modulation[ch]` をループで `controlChange(ch, 1, v)` する。

> **✅ 対応済み** `sf_synth.cpp:955-957` — `expr.modulation[ch]` をループで `controlChange(ch, 1, v)` に渡すコードを追加。

#### 3. CC65（Portamento On）が `controlChange` で無視される

`renderToWav` は `expr.portamentoOn` を `controlChange(ch, 65, v)` に渡すが、内部で破棄される。`noteOn` のポルタメント処理が機能しない。

**修正提案:** `case 65: ch.portamentoOn = value; break;`

> **✅ 対応済み** `sf_synth.cpp:500` — `case 65: break;` を `case 65: ch.portamentoOn = value; break;` に変更。

#### 4. `--pitch` 使用時の noteOff 不一致

noteOn では pitchShift を適用するが noteOff では適用していない。

**修正提案:** noteOff でも同じ `pitchShift` を適用する。

#### 5. `tickForTime` 逆変換が不正確

秒→tick の線形補間がなく、テンポ変更後の CC/PC/SysEx タイミングがずれる。

**修正提案:** `tickToSeconds` の逆関数を実装し、区間ごとに BPM で補間する。

#### 6. SC-8850 モデル検出のオフセット誤り

Identity Reply でモデル ID のオフセットが誤っている（`midi_file.cpp` L227–231）。

**修正提案:** `addr1==0 && addr2==0 && addr3==0` かつ `data[0..2] == {0x54,0x00,0x14}` で判定。

#### 7. SysEx 後の Running Status 汚染

SysEx（0xF0）処理後、`runningStatus = 0xF0` のまま残る。MIDI 仕様では SysEx は Running Status をクリアする。

**修正提案:** SysEx 処理後に `runningStatus = 0`。

#### 8. RPN 1（Fine Tune）の Data Entry 解釈が誤り

RPN 1 は 14bit（CC6+CC38、中心 8192）。MIDI パーサが RPN を一切記録していない（CC98–101, CC6/38 未対応）。

#### 9. トラック終端で noteOn 残存

`extractNotes` はトラック末尾で鳴り続けるノートをフラッシュしない。

#### 10. 同一チャンネル・同一ノートのポリフォニー追跡不可

`noteOn[channel][note]` は 1 エントリのみ。

#### 11. `renderToWavPerChannel` が CC/エフェクト未適用

チャンネル分割出力は program/bank 初期化のみ。

#### 12. SF2 Modulator リスト未処理

`presetBags`/`instBags` の `modNdx` を無視。

- Variable-length delta、Running Status、NoteOn vel=0 as NoteOff

### 不具合・バグの指摘と修正提案（WAV reproduction focused）

#### 1. 【致命的】`tickForTime` により CC/表現イベントがブロックに適用されない

```889:902:cpp/src/sf_synth.cpp
            auto tickForTime = [&](double sec) -> int64_t {
                int64_t tick = 0;
                // ...
                for (auto& [t, bpm] : tmap) {
                    double thisTime = midi.tickToSeconds(t);
                    if (thisTime >= sec) break;
                    tick = t;
                }
                return tick;
            };
            blockTickStart = tickForTime(blockSecStart);
            blockTickEnd = tickForTime(blockSecEnd);
```

- テンポ変更点の tick のみ返し、**秒→tick の線形補間がない**
- 単一テンポ MIDI では `blockTickStart == blockTickEnd == 0` が常に成立
- 適用条件 `t >= blockTickStart && t < blockTickEnd`（L912, L928 等）が **常に false** → Volume/Expression/Pan/Sustain/Pitch Bend/PC がレンダ中に無視される
- **修正:** `tickToSeconds` の逆関数（区間 BPM 補間）を実装。`blockTickEnd` はブロック終端 tick（排他的）を正確に計算。`blockTickStart == blockTickEnd` のときは `blockTickEnd + 1` 等で空区間を回避

#### 2. 【致命的】`processVoice` のピッチ進行が `pitchRatio` を使わない

```714:731:cpp/src/sf_synth.cpp
        double effectivePitchRatio = v.basePitchRatio;
        if (v.portamentoProgress < 1.0) {
            // ... 補間 ...
            effectivePitchRatio = v.basePitchRatio + (v.pitchRatio - v.basePitchRatio) * t;
        }
        v.position += effectivePitchRatio * std::pow(2.0, vibratoShift / 12.0);
```

- ポルタメントなし（`portamentoProgress == 1.0`）では `basePitchRatio`（既定 **1.0**）のみ使用
- `startVoice`（L274–286）は `v.pitchRatio` を正しく計算するが **`basePitchRatio` にコピーしない**
- メロディはルートキー付近の固定ピッチで再生され、**音程が MIDI と一致しない**
- ドラムは `pitchRatio ≈ sampleRate/44100 ≈ 1.0` のため「リズムはそれなり、メロディがおかしい」という症状と一致
- **修正:** `startVoice` で `v.basePitchRatio = v.pitchRatio`、または `else` で `effectivePitchRatio = v.pitchRatio`

#### 3. チャンネル初期化が program/bank のみで CC 状態を復元しない

```824:851:cpp/src/sf_synth.cpp
    for (int ch = 0; ch < 16; ch++) {
        // firstNoteTick 以前の PC/Bank のみ探索
        // volume/expression/pan/sustain/pitchBend は未設定
        programChange(ch, m_channels[ch].program, m_channels[ch].bank);
    }
```

- `ChannelState` 既定値（volume=100, expression=127, pan=64）のまま開始
- 最初のノート以前の CC7/CC11/CC10/CC64 等が無視される
- `MidiExpression::getValueAtTick`（`midi_file.h` L125–128）はあるが未使用
- **修正:** 各 ch の `firstNoteTick` で `getValueAtTick` により全 CC を初期化してから `programChange`

#### 4. CC1（Modulation）が `renderToWav` ループに存在しない

- パース: `midi_file.cpp` L177
- 合成: `controlChange` case 1（`sf_synth.cpp` L477）は実装済み、`processVoice`（L581–583）も参照
- レンダループ（L926–957）に **modulation 適用ループがない**（Review #1 の指摘を再確認）

#### 5. CC0 単独変更でプリセット再構築されない（Review #1 継続）

- L909 `hasBankSelectInBlock` を設定するが L920 の `needProgramChange` とは連動しない
- Bank MSB のみ変更時 `buildPresetZones` が呼ばれない

#### 6. noteOn/Off のブロック量子化と noteOff ロジックの非対称

**メインパス（`renderToWav`）:**
- noteOn: ブロック内サンプル位置で発火（L962–963）— ブロック先頭に量子化
- noteOff: `noteEnd <= pos`（L980）— **ブロック開始時のみ判定**、最大 ~23ms（1024/44100）遅延

### 不具合・バグ

#### 1. 【高】noteOn/Off がブロック境界に量子化 — 「ノイズ・ずれ・伸び」の主因

```1026:1050:cpp/src/sf_synth.cpp
        // Note On: start new notes in this block
        for (auto& n : sorted) {
            int noteStart = (int)(midi.tickToSeconds(n.startTime, n.track) * m_sampleRate);
            if (noteStart >= pos && noteStart < blockEnd) {
                // ...
                noteOn(n.channel, shiftedNote, n.velocity);
            }
        }
        // Note Off
        for (auto& a : active) {
            int noteEnd = (int)(midi.tickToSeconds(a.note->endTime, a.note->track) * m_sampleRate);
            if (noteEnd <= pos) {
                noteOff(a.note->channel, shiftedNote);
```

- **noteOn:** ブロック内の任意 tick でも **ブロック先頭**で発火 → 最大 ~23ms **早鳴り**
- **noteOff:** `noteEnd <= pos` は **次ブロック開始時**にのみ判定 → 最大 ~23ms **遅延リリース**
- 短いノート（スタッカート、ドラム）は特に「濁る／伸びる／不自然」
- Per-channel パスは noteOff が `noteEnd >= pos && noteEnd < blockEnd`（`1290-1293`）と**非対称**

**修正提案:** ブロック内サンプルオフセットで noteOn/Off。またはイベント駆動レンダラへ統一。

#### 2. 【高】RPN/CC6/CC38/CC98–101 が MIDI からレンダに届かない

- `midi_file.cpp` の CC switch に RPN 系が**一切ない**（`175-188`）

#### 3. 【高】CC 初期化で per-channel エフェクト深さの取り違え

### 不具合・バグの指摘と修正提案

#### 1. 【致命的】`sampleType` ループ判定が SF2 仕様と不一致

```216:216:cpp/src/sf_synth.cpp
 z.loop = (s.sampleType & 0x1000) != 0;
```

SF2 仕様では `sampleType`（16bit）の**上位バイト（bits 8–15）がループモード**。

| 典型値 | 意味 | `(x & 0x1000)` |
|--------|------|----------------|
| `0x0101` | mono + loop continuously | **false**（bit 12 未セット） |
| `0x0001` | mono + no loop | false |

GM ピアノの `0x0101` 等は**ループありなのに判定 false** → ほぼ全サステイン楽器でループ無効。

`processVoice` ではループ OFF 時、`sampleEnd` 到達でボイス強制終了:

```653:663:cpp/src/sf_synth.cpp
 if (v.loop && v.loopEnd > v.loopStart) {
 // ... loop ...
 } else if (idx >= (int)v.sampleEnd) {
 v.active = false;
 break;
 }
```

**修正提案:**

```cpp
int loopMode = (s.sampleType >> 8) & 0xFF;
z.loop = (loopMode == 1 || loopMode == 3); // 1=continuous, 3=loop until release
```

FluidSynth 等と同様に mode 3 は「全サンプル再生後ループ」も実装するとより忠実。

```128:128:cpp/src/sf_synth.cpp
 case 24: def.fineTune = a / 100.0; break; // セント→半音
```

```209:210:cpp/src/sf_synth.cpp
 z.rootKey += z.fineTune / 100.0; // fineTune は既に半音単位なのに再度 /100
```

**修正:** `z.rootKey += z.fineTune;`

### 新規/継続の重要指摘

#### 1. 【P1】`sm24` 合成が 24bit PCM として危険

`soundfont.cpp:117-121`:

```cpp
m_sampleData[i] = (m_sampleData[i] << 8) | sm24Data[i];
```

問題点:

- `m_sampleData[i]` は `int32_t` だが、元は `int16_t` 相当の上位 16bit。負値を左シフトする処理は C++ では危険で、処理系依存/未定義寄りの挙動になりうる。
- `sm24Data[i]` は下位 8bit だが、符号拡張・上位/下位の結合・正規化の扱いが明確でない。
- `processVoice()` は常に `/ 8388608.0` で読む（`sf_synth.cpp:749`）。`sm24` がない 16bit SF2 でも値が 256 倍されていない場合、音量が極端に小さくなる。現コードでは `smpl` 読み込み時に 16bit 値を `int32_t` へ memcpy しているため、16bit-only SF2 のスケールが壊れる可能性が高い。

**音質影響:** 16bit SF2 で音量低下、24bit SF2 で歪み/ノイズ/符号反転が起こりうる。R#18 の「高品質化」候補が、逆に音質劣化の入口になっている可能性。

**推奨:** 内部表現を明確にする。例えば `sm24` なしなら `int32_t(sample16) << 8`、`sm24` ありなら `((int32_t)sample16 << 8) | lsb` とし、正規化は常に 24bit 基準にする。符号付き値の左シフトは明示的に安全な型変換で扱う。

#### 2. 【P1】`delayVolEnv` / `holdVolEnv` は generator を読んでも zone に反映されていない

`GenState` に `delayVolEnv/holdVolEnv` はある（`sf_synth.cpp:115-116`）が、`applyGen()` は `st.delayVolEnv` / `st.holdVolEnv` に代入しているだけ（`:134-135`）。`mergeState()` はそれらを `r.zone.delayVolEnv` / `r.zone.holdVolEnv` へコピーしない（`:169-218`）。

結果として `startVoice()` は `zone.delayVolEnv` / `zone.holdVolEnv` を読む（`:401-406`）が、ほぼ常に初期値 `0.0` のままになる。

**音質影響:** Review #18 で期待した VolEnv delay/hold 改善は実効していない。さらに「実装済みに見えるが効かない」状態なので、後続調査を誤らせる。

#### 3. 【P1】`scaleTuning` は開始時に未初期化相当の値を使い、その後の live update では消える

`startVoice()` では `v.scaleTuning` を pitchRatio 計算に使ったあと（`sf_synth.cpp:378-381`）、`v.scaleTuning = zone.scaleTuning` を設定している（`:408-409`）。この順序だと初回 pitchRatio 計算では `SF2Voice{}` の既定値 `1.0` を使い、zone の値が反映されない。

さらに `processVoice()` の live pitch update は scale tuning を含めていない（`:656-664`）。したがって pitch bend/tuning 更新時には scale tuning が失われる。

**音質影響:** scaleTuning 対応は現状ほぼ効かない。音程や民族楽器/特殊音律系プリセットで差が出る可能性。

#### 4. 【P1】SF2 send は保持されるが FX bus に届かない

`case 15/16` で `zone.reverbSend/chorusSend` に値を入れている（`sf_synth.cpp:152-154`）が、`mergeState()` は `reverbSend/chorusSend` を合成しない。さらに `startVoice()` / `processVoice()` / `applyFxSegment()` は voice の send を持たず、FX は channel CC/GS SysEx だけで決まる（`:1231-1274`）。

**音質影響:** R#18 の「SF2 プリセット内蔵の空間設計」は未反映。現状ではフィールド追加だけで音は変わらない。

#### 5. 【P2】Envelope が指数カーブから線形カーブへ変わっている

`SF2Voice` は `attackCoeff/decayCoeff/releaseCoeff` から `attackRate/decayRate/releaseRate` へ変わり（`sf_synth.h:32-38`）、`processVoice()` は `envLevel += attackRate` / `envLevel -= decayRate` の線形処理になっている（`sf_synth.cpp:681-705`）。

**音質影響:** SF2 の時間値を取り込む方向としては前進だが、楽器の立ち上がり/減衰の質感が変わる。ユーザーが以前「音が劣化した」と感じたような、アタックの硬さ/リリースの不自然さにつながる可能性がある。

#### 6. 【P2】`pmod/imod` 未実装のままなので、音質改善の本丸は未着手

`modNdx` は bag から読めているが、`pmod/imod` を保存しないため、以下が未再現:

- velocity to attenuation / filter
- CC1 to vibLFO pitch
- channel pressure / poly pressure
- mod envelope to pitch/filter
- instrument/preset 固有の modulation routing

**音質影響:** 強弱や楽器ごとの表情が GM 的な線形近似に留まる。Review #18 の最重要改善候補は継続。

### Review #19 結論

Review #18 の実装は一部進んでいるが、現時点では **「音質改善として有効に鳴る実装」より「途中まで足場を作った実装」** が多い。

特に危険なのは `sm24` と envelope/scaleTuning まわり。`sm24` は 16bit SF2 の音量スケールを壊す可能性があり、scaleTuning と VolEnv delay/hold はコード上は実装済みに見えるが実際には届いていない。

次に修正するなら、優先順位は以下:

1. **`sm24` 内部表現と正規化の修正** — 16bit SF2 の音量が壊れていないか最優先確認
2. **`delayVolEnv/holdVolEnv/scaleTuning/reverbSend/chorusSend` の mergeState 接続** — フィールド追加を実効化
3. **scaleTuning の計算順序と live pitch update 反映**
4. **SF2 send を per-voice/per-channel FX bus へ接続**
5. **`pmod/imod` 最小セット実装**
6. **ステレオサンプルペア対応**

短期の音質改善としては、`pmod/imod` より前に **現在の部分実装の破綻を直す** 方が安全。ここを整えないままステレオやモジュレータへ進むと、改善点と劣化点が混ざって試聴判断が難しくなる。

### うねり音の最有力原因

#### 1. 【P1】CC1 が pitch / filter / tremolo を同時に揺らしている

現行コードでは、CC1 modulation が非ドラム全ボイスの `vibratoDepth` になる。

- `v.vibratoDepth = modulation / 127.0 * 2.0`（`sf_synth.cpp:442`, `:696-699`）
- LFO は全ボイス 5Hz 固定（`:716-721`）
- 同じ LFO が pitch に入る（`:887-908`）
- 同じ LFO が filter cutoff に入る（`:833-842`）
- 同じ LFO が tremolo に入る（`:861-875`）

つまり CC1 が大きい箇所では、**ビブラート + ワウ + トレモロ** が同時に発生する。ハーモニカ、オルガン、ストリングス、シンセ、ティンパニ的な持続/減衰音では、これは「うねり」「揺れすぎ」「酔うような音」として聞こえやすい。

特に `01COMIN&.MID` は CC1 が 872 件あり、値も 0-127 全域を使う。現行実装では MIDI 側の表情指定が過剰な音色変調に増幅される可能性が高い。

**SF2適合の観点:** CC1 の効果は本来 `pmod/imod` と既定モジュレータで決まる。現行の「全楽器一律 2 semitone + filter + tremolo」は SF2準拠ではない。

#### 2. 【P1】ステレオ sampleLink/sampleType 未使用により、ペアサンプルが mono layer として鳴る

`sampleLink/sampleType` は `soundfont.cpp:192-193` で読んでいるが、再生側では使われていない。`processVoice()` は `sampleIndex` の単一サンプルを読み、pan で左右へ配るだけ（`sf_synth.cpp:802-882`）。

SF2 のステレオサンプルは、本来 left/right linked sample を一組として扱うべき。これを mono layer として複数 zone 再生すると、以下が起きる。

- 左右ペアが別々に同時発音し、音量や位相が不自然になる
- 同じ楽器の近いサンプルが重なり、微妙なピッチ差やループ差で beating が出る
- ハーモニカ/ストリングス/シンバル/ティンパニ系の持続音で「うねり」として聞こえる

**SF2適合の観点:** ステレオサンプル未対応は、SF2音質を引き出せていない大きな要因。

#### 3. 【P1】`pmod/imod` 未実装のため、楽器固有のモジュレーション量を無視している

`pbag/ibag` の `modNdx` は読む（`soundfont.cpp:141-143`, `:165-168`）が、`pmod/imod` chunk をパースしない。これにより、SF2 が本来持つ以下の制御が失われる。

- CC1 がどの程度 pitch に効くか
- CC1 が filter に効くか、効かないか
- velocity が filter/attenuation にどう効くか
- ModEnv/LFO が pitch/filter/volume にどう効くか

現行コードはその穴を独自実装で埋めようとしているが、楽器ごとの差が出ない。結果として、ハーモニカのように元々ビブラート感のある音色へ、さらに一律LFOが足され、うねりが強調される可能性がある。

#### 4. 【P2】filter cutoff の変調が音色の“ワウ”を作っている

`modulatedFc` は CC1由来LFOと ModEnv で揺れる（`sf_synth.cpp:833-842`）。フィルタ処理は biquad low-pass（`:843-859`）。

フィルタが有効な zone で CC1 が動くと、音量やピッチだけでなく倍音量も周期的に動く。これは耳には「ハーモニカっぽいワウ」「うねるティンパニ胴鳴り」に聞こえやすい。

**疑い:** ユーザー報告の「ティンパニやハーモニカっぽい音でうねり音」は、単なる音程ビブラートより **filter/tremolo 併用** の可能性が高い。

#### 5. 【P2】ループ境界補間が loop-aware ではない

loop は `idx >= loopEnd` で `loopStart` 側に戻す（`sf_synth.cpp:779-795`）が、補間 tap の `getSample(idx + 1/2)` は `sampleEnd` 側へ clamp する（`:802-830`）。loopEnd 付近では、本来 loopStart へ回り込むべき補間が回り込まない。

**音質影響:** sustained sample のループ境界で周期的なクリック/濁り/揺れが出る。これは5Hz LFOとは別の、サンプル周期由来のうねりとして聞こえる可能性がある。

#### 6. 【P2】Scale tuning は開始時も live update も不完全

`startVoice()` の pitchRatio 計算では `v.scaleTuning` を使うが、`v.scaleTuning = zone.scaleTuning` はその後に実行される（`sf_synth.cpp:384-423`）。さらに `processVoice()` の live pitch update は scaleTuning を含めていない（`:701-709`）。

**音質影響:** scale tuning 実装は有効に見えるが、実際の音程には安定して反映されない。音程差を持つ layer が重なると beating の原因になる。

#### 7. 【P2】SF2 send は voice に入るが FX へ届かない

`v.reverbSend` / `v.chorusSend` は設定される（`sf_synth.cpp:446-448`）が、FX mix は channel CC/GS SysEx の平均だけを見る（`:1298-1341`）。voice send は未使用。

**音質影響:** SF2プリセット固有の chorus/reverb 設計が反映されない。逆に MIDI CC93/GS chorus が全体へ強くかかる場合、全パートが同じ揺れ方をし、うねりが目立つ。

### 優先度付き改善提案

1. **CC1 の適用先を一旦 pitch のみに絞る、または tremolo/filter 併用を無効化して A/B 試聴**
   - うねり原因の切り分けとして最短。
   - `01COMIN&.MID` は CC1 が多いため効果確認しやすい。

2. **`pmod/imod` の最小実装**
   - まず SF2既定モジュレータだけでよい。
   - CC1 → pitch depth をSF2仕様寄りにし、filter/tremoloへの勝手な流用を止める。

3. **linked stereo sample 対応**
   - `sampleType` と `sampleLink` に基づき、left/right を1 voiceの stereo sample として読む。
   - mono layer の重複発音による beating を減らせる可能性が高い。

4. **loop-aware interpolation**
   - 補間 tap が loopEnd を超えたら loopStart 側を読む。
   - 持続音の周期的なザラつき/うねり改善に効く。

5. **scaleTuning の適用順序と live update 修正**
   - layer間の微妙な音程差を抑える。

6. **SF2 send を FX bus に接続**
   - ただし chorus はうねりを増やす可能性があるので、まず send量をログ/制限しながら導入する。

### Review #20 結論

SF2適合度は現状 **35-45%程度**。基本的なSF2サンプルプレイヤーとしては動くが、SF2音源の性能を100%引き出せてはいない。

ユーザー報告の「ティンパニやハーモニカっぽい音でうねり音」は、現行コード上は **CC1 modulation を pitch/filter/tremolo に同時適用していること**が最有力。さらに、ステレオサンプル未対応、loop補間の非loop-aware、`pmod/imod`未実装が重なると、持続音でうねりが強く聞こえる可能性が高い。

短期的には、まず **CC1由来の filter/tremolo 変調を切って試聴**するのがよい。ここでうねりが大きく減るなら、次は `pmod/imod` と stereo sample 対応へ進むべき。

### CC再現性レビュー

#### 1. 【P1】CC1 modulation の再現がSF2準拠ではなく、うねり音の主因になりうる

現行の CC1 は全楽器一律で最大2半音、5Hz固定ビブラートになる。SF2では、CC1の効果量・宛先・カーブは既定モジュレータおよび `pmod/imod` に依存する。現行実装はこれを無視している。

**影響:**

- ハーモニカ音色に元々含まれる揺れへさらにピッチLFOが乗る
- ティンパニ/低音持続音で胴鳴りの周期変動のように聞こえる
- CC1が多いMIDIでは、実機より大げさな揺れになる

`01COMIN&.MID` は CC1 が 872 件あり、値域も 0-127。うねり症状と相関が高い。

#### 2. 【P1】CC91/93 と SF2 chorus/reverb send の扱いが全体処理で、個別楽器の空間設計になっていない

FX適用は block segment 単位で、全体の `segL/segR` に対して reverb/chorus/delay をかける（`sf_synth.cpp:1325-1386`）。SF2 `reverbSend/chorusSend` は active voices から平均/最大寄りに混ぜられるが、voice別・channel別 bus ではない。

**影響:**

- 特定楽器の chorus send が、同時に鳴る全パートへ広がる
- chorus は遅延LFO系なので、うねりや位相揺れを増やす
- 「ハーモニカっぽい揺れ」が全体に薄く乗る可能性がある

CC再現性としては「CC91/93を拾う」段階は達成しているが、ミックス構造は実機/ SF2 仕様と違う。

#### 3. 【P2】CC7/CC11 はサンプル精度で処理されるが、音量カーブは線形寄り

CC7/CC11 はイベントとして収集され（`sf_synth.cpp:1236-1239`）、`controlChange()` で状態更新される（`:614-616`）。`processVoice()` では `volume / 127.0` と `expression / 127.0` をそのまま掛ける（`:898-902`）。

**影響:** MIDIイベントの時刻再現は良いが、実機/GM音源的なdBカーブとは違う可能性がある。音量のうねりというより、パートバランスの違和感として出る。

#### 4. 【P2】RPN/ピッチベンドは処理されるが、scaleTuningがlive updateに入らない

pitch bend / tuning は live update される（`sf_synth.cpp:727-735`）。しかし scaleTuning は `startVoice()` の設定順にも問題があり、live update側にも含まれない。

**影響:** layer間でわずかな音程差が残ると、beating のようなうねりになる。CC1ほど直接ではないが、複数zone音色では無視しにくい。

#### 5. 【P2】Sustain CC64は状態管理されるが、同音重なり/voice releaseとの組み合わせは要注意

CC64は保持中voiceを release へ送る（`sf_synth.cpp:630-643`）。一方で、MIDI note抽出は同一note重なりを stack 処理している（`midi_file.cpp:300-319`）。曲によっては同音重なりの対応順が実機と異なり、サステイン中の濁り・残響的なうねりにつながる可能性がある。

---

### うねり音の原因候補ランキング

| 優先度 | 原因候補 | 根拠 | コメント |
|--------|----------|------|----------|
| P1 | **CC1固定5Hz/最大2半音ビブラート** | `sf_synth.cpp:722-748`, `:914-925` | 最有力。ハーモニカ/持続音/低音で目立つ |
| P1 | **chorusを全体segへ適用** | `sf_synth.cpp:1349-1368` | CC93/SF2 chorus send が全体の揺れになる |
| P1 | **stereo sample未対応** | `sampleLink/sampleType` 未使用 | linked L/R や近接layerの位相差で beating |
| P2 | **loop補間がloop-awareでない** | `getSample()` が sample範囲 clamp | 持続音の周期ノイズ |
| P2 | **scaleTuning/live pitch不整合** | pitch update に scaleTuningなし | layer間 beating の温床 |
| P2 | **ModEnv/Filterの独自近似** | `modEnvToFilterFc` 等を簡易適用 | 音色変化がSF2設計と違う |

---

### SF2/CC再現性を上げる実装優先度

1. **CC1をSF2既定モジュレータ相当に寄せる**
   - まず最大深度を下げる、または `pmod/imod` 未実装中は pitch only / 小さめ深度に制限。
   - A/B確認では CC1無効版、CC1深度25%版、現行版を比較すると原因が切り分けやすい。

2. **chorus/reverbを全体seg処理からsend bus処理へ分離**
   - 少なくとも chorus は channel/voice send 対象音だけにかける。
   - 現状は1つの揺れエフェクトが全パートに乗る。

3. **`pmod/imod` 最小実装**
   - CC1、velocity、channel pressure、mod LFO depthだけでも効果が大きい。
   - 独自ハードコードよりSF2の意図に近づく。

4. **linked stereo sample対応**
   - left/right sampleを1つのstereo voiceとして扱う。
   - mono重複や位相差由来のうねりを減らす。

5. **loop-aware interpolation**
   - `idx+1`, `idx+2` が loopEnd を越えたら loopStart側を読む。
   - 持続音の周期的な荒れを低減。

6. **CCカーブの実機寄せ**
   - CC7/11を線形gainではなくdB寄りにするか、少なくとも比較モードを用意する。

---

### Review #21 結論

SF2適合度はまだ **40%前後**。Review #20から一部改善は見えるが、`pmod/imod`未対応、stereo sample未対応、`sm24`未対応、chorus/reverb bus未分離が残っており、SF2の性能を十分には引き出せていない。

CC再現性は「イベントを拾ってサンプル精度で適用する」面では改善しているが、**CCが音源内で何にどれだけ効くか**はSF2仕様とズレている。特に CC1 は全楽器一律の固定5Hz/最大2半音ビブラートで、うねり音の最有力原因。

次の切り分けは、**CC1深度を下げる/無効化する試聴**と、**chorusを一時的に切る試聴**が最短。そこでうねりが減るなら、`pmod/imod` と effect send bus の実装優先度が確定する。

**対象:** `D:/tmp/wav/MIDI1.0.pdf`, `D:/tmp/wav/sfspec24.pdf` を根拠にした MIDI/SF2 適合度レビュー。

---

### 仕様書から見た主要前提

#### MIDI 1.0 / SMF

`MIDI1.0.pdf` の Standard MIDI File 仕様では、MTrk event は `<delta-time> <event>` で、delta-time は可変長表記かつ常に存在する。MIDI channel event では running status が使えるが、各 MTrk chunk の最初の event は status を持つ必要がある。また、SysEx event は `F0 <length> <data>` または `F7 <length> <data>` 形式で、`length` は F0/F7 と length 自身を含まない。SysEx と meta event は running status を無効化する。

現行コードはこの基本にかなり沿っている。

- delta-time は `readVarLen()` で読まれ、tick に加算される（`midi_file.cpp:186-188`）
- running status は channel event で保持される（`:189-190`）
- meta event と SysEx で running status をクリアする（`:193`, `:217`）
- SysEx は `F0/F7` の後に可変長 length を読む（`:211-219`）

ただし、`midi_file.cpp:368-472` に古い SysEx 解析ブロックが残っており、現在の先行 `F0/F7` 処理後には実質到達しない死コードになっている。仕様適合そのものより保守リスク。

#### SoundFont 2.04 / 2.01 technical model

`sfspec24.pdf` では、SoundFont は generator と modulator を中心に設計されている。特に以下が重要。

- `PMOD` / `IMOD` は zone modulator の必須 sub-chunk で、各 record は `sfModSrcOper`, `sfModDestOper`, `modAmount`, `sfModAmtSrcOper`, `sfModTransOper` を持つ。
- `IMOD` は default modulator を置換でき、`PMOD` は `IMOD` に対して相対的に加減できる。
- default modulator には CC1 -> Vibrato LFO Pitch Depth、CC7/CC11 -> Initial Attenuation、CC10 -> Pan、CC91/CC93 -> Reverb/Chorus Send が定義されている。
- CC1 の default amount は **50 cents/max excursion**。
- stereo sample では `sfSampleType` が left/right を示し、`wSampleLink` が相手 sample header index を示す。left/right pair は同期再生され、pitch は right sample の generator によって制御される。
- `sm24` は `smpl` の各 16bit sample に対応する下位 8bit を持つ。`smpl` がない場合や SF2.04 未満では無視する。

---

### Review #21 からの更新点

現行コードでは、Review #21 時点より CC1 の深度が改善されている。

- `v.vibratoDepth = modulation / 127.0 * 0.5`（`sf_synth.cpp:465-468`, `:722-737`）

これは SF2 default modulator の **50 cents** に近く、Review #21 の「最大2半音」よりは仕様に寄った。ただし、まだ `pmod/imod` による上書き・加算、`freqVibLFO` / `delayVibLFO`、channel pressure は反映されていない。

---

### 指摘一覧

#### 1. 【P1】`pmod/imod` 未実装のため、SF2適合度の上限が低い

仕様書では `PMOD` / `IMOD` が modulator list として定義され、default modulator の置換・加算規則も明記されている。しかし現行の `parsePDTA()` は `phdr/pbag/pgen/inst/ibag/igen/shdr` を読むだけで、`pmod/imod` chunk を保存しない（`soundfont.cpp:113-200`）。`pbag/ibag` の `modNdx` は読む（`:141-143`, `:165-168`）が未使用。

**影響:** CC1、velocity、CC7/11、CC10、CC91/93、channel pressure の効果が SF2 bank の設計通りにならない。SF2の「性能を100%引き出す」にはここが最大の未達。

#### 2. 【P1】CC1 は50 centsへ近づいたが、VibLFO仕様としては未完成

仕様書の default modulator は CC1 -> Vibrato LFO Pitch Depth、amount 50 cents/max excursion。現行コードの `0.5` semitone は量としてはかなり近い。

しかし、現行は LFO 周波数を 5Hz 固定にしている（`sf_synth.cpp:742-748`）。仕様では `freqVibLFO` generator が存在し、`delayVibLFO` も定義される。現行コードは generator 23/24 を実装していないため、SF2が意図したビブラート速度・遅延を再現できない。

**うねり音への影響:** 深度は改善されたが、すべての楽器で同じ5Hz揺れになるため、ハーモニカ/ティンパニ系の持続音ではまだ不自然な周期感が残る可能性がある。

#### 3. 【P1】stereo sample の仕様未対応が、位相うねり/beating の大きな原因候補

仕様書では left/right sample は `wSampleLink` で関連付けられ、同期再生されるべきとされている。現行コードは `sampleLink/sampleType` を読む（`soundfont.cpp:192-193`）が、再生側で使わない。`processVoice()` は単一 sample を mono として読み、pan で左右へ配る（`sf_synth.cpp:836-909`）。

**影響:** stereo pair を別zone/別voiceとして鳴らすと、位相差・ループ差・微妙な pitch 差が beating になりやすい。ユーザー報告の「うねり」は CC1 だけでなく、stereo未対応からも起こりうる。

#### 4. 【P1】CC7/CC11 はイベント再現は良いが、SF2 default modulator の凹カーブではない

仕様書では CC7/CC11 は Initial Attenuation へ入り、Negative Unipolar source で concave mapping とされる。現行コードは `volume / 127.0` と `expression / 127.0` を直接乗算する（`sf_synth.cpp:898-902`）。

**影響:** CCの時刻再現はサンプル精度で良いが、音量カーブはSF2仕様/実機の聴感と違う。強弱の膨らみ方が不自然になり、揺れやまとまりの悪さとして聞こえる可能性がある。

#### 5. 【P2】CC91/93 と SF2 send の扱いが仕様の「send」ではなく、全体FX mixになっている

仕様書では chorus/reverb send は「note の audio output を effects processor へ送る量」で、dry量には影響しない。default modulator でも CC91/93 は Reverb/Chorus Send に加算される。

現行コードは active voice の `reverbSend/chorusSend` や CC値を見て、セグメント全体 `segL/segR` に reverb/chorus をかける（`sf_synth.cpp:1325-1368`）。これは send bus ではなく、すでに混ざった全体音に後段で処理する形。

**影響:** 特定パートの chorus send が全体に揺れを足す。うねり音の二次原因として有力。

#### 6. 【P2】`sm24` 未対応により SF2.04 の24bit性能は使えていない

仕様書では `sm24` が `smpl` の下位8bit counterpart として定義される。現行 `parseSDTA()` は `smpl` だけを読み、`sm24` を無視する（`soundfont.cpp:91-110`）。

**影響:** 16bit SF2では問題ないが、24bit SF2では微小音・残響尾・弱音の解像度を捨てる。SF2.04性能を100%引き出すという観点では未達。

#### 7. 【P2】Generator番号・意味の保守リスクが残る

`soundfont.h` の `SF2GenOper` enum は `keyRange=5`, `velRange=6`, `initialFilterFc=43` など古い/誤った値のまま（`soundfont.h:25-61`）。実際の `applyGen()` は数値 switch で `8=initialFilterFc`, `43=keyRange` などを使う（`sf_synth.cpp:123-166`）。

**影響:** 現状は switch が効いていても、enumを使う修正で再びSF2 generator番号を壊す危険がある。仕様書ベースの単体テストが必要。

#### 8. 【P3】SMF/SysExは概ね仕様に合うが、死コードとFormat 2精度は残課題

MIDI1.0資料のSMF記述に対して、running status・SysEx length・meta/SysEx後のrunning status clearは概ね合っている。ただし古いSysEx解析ブロックが残っている。さらに Format 2 のような独立トラック時間軸の厳密性は、レンダー側 `tickToSampleOffset(tick)` が track 0 を使う箇所などに注意が必要。

### うねり音に対する資料ベースの結論

最有力は引き続き **modulation / chorus / stereo未対応** の組み合わせ。

ただし Review #21 から見ると、CC1深度は仕様に近くなったため、単純な「深すぎるビブラート」だけでは説明しきれない。今後の優先順位は以下。

1. **`freqVibLFO` / `delayVibLFO` 対応**  
   5Hz固定をやめ、SF2が指定するLFO周期・遅延に従う。

2. **`pmod/imod` 最小対応**  
   CC1 default modulator を bank 側で置換/無効化できるようにする。

3. **chorus/reverb send bus 分離**  
   全体FXではなく、send対象音だけをwet busへ送る。

4. **stereo sample pair対応**  
   linked left/right sampleを同期再生する。

5. **CC7/11/91/93 の default modulator カーブ対応**  
   線形gainや全体mixではなく、仕様の attenuation/send model へ寄せる。

### Review #22 結論

資料に照らすと、現行コードは MIDI 1.0 のSMF基本処理にはかなり近い。一方、SoundFont 2.04/2.01 technical model への適合はまだ中程度で、SF2の性能を100%引き出せてはいない。

特に `pmod/imod`、stereo sample、VibLFO generator、CC7/11/91/93 default modulator、send bus が未達。うねり音については、CC1深度は改善されたが、5Hz固定LFO・chorus全体処理・stereo未対応が残っているため、まだ再発しうる。

次の修正で最も仕様効果が高いのは、**`pmod/imod` 最小実装 + VibLFO generator対応 + stereo sample対応**。この3つが入らない限り、SF2適合度は50%台を大きく超えにくい。
